## 1. 产品概述

### 1.1 产品背景与问题陈述

在线教育市场规模持续扩大，但现有平台普遍面临以下核心痛点：

| 痛点 | 具体表现 |
|------|---------|
| **千人一面** | 学习资源统一推送，无法匹配个体认知水平与学习风格 |
| **路径僵化** | 学习路线固定，学习者无法根据自身进度灵活调整 |
| **反馈滞后** | 学习效果评估依赖期末考试或人工批改，无法实时干预 |
| **资源碎片化** | 学习者需要在多个平台间跳转搜集资料，缺少整合式学习体验 |

大语言模型（LLM）的成熟为上述问题提供了新的解决路径。智伴（ZhiBan）利用多智能体协作架构，将用户画像分析、个性化资源生成、学习路径规划和效果评估整合为一体化闭环，实现"千人千面"的自适应学习体验。

### 1.2 目标用户群体

- **在校学生**（高中 / 大学）：需要针对具体科目进行系统化学习或备考的主力用户群
- **职场学习者**：在职充电、考证、技能转型的成人学习者
- **教师 / 培训师**：希望借助 AI 工具高效备课、生成教学资源的教育工作者

> ⚠️ **假设**：MVP 阶段优先支持大学生群体，科目范围限定在计算机科学和英语两个品类，后续逐步扩展。待确认具体首发科目范围。

### 1.3 产品核心价值主张

**"你的 AI 学伴，懂你的学习节奏"**

| 维度 | 价值 |
|------|------|
| 个性化 | 基于用户画像动态生成匹配其水平与偏好的学习资源 |
| 自适应 | 学习路径根据实时评估结果自动调整，避免"太难放弃 / 太简单无聊" |
| 一站式 | 资源生成、路径规划、效果评估闭环完成，无需多平台切换 |
| 持续进化 | 系统随用户学习数据积累，推荐精准度持续提升 |

## 2. 用户角色与使用场景

### 2.1 Persona 1：小明 — 大二计算机系学生

| 属性 | 描述 |
|------|------|
| 年龄 | 20 岁 |
| 目标 | 系统学习数据结构与算法，备战实习面试 |
| 痛点 | 教材太厚不知从哪开始；刷题没有方向，不知道薄弱环节在哪 |
| 技术素养 | 中等，能使用基本开发工具 |

**典型 User Story：**

- **US-1.1**：作为一名计算机专业学生，我希望系统能根据我当前的算法掌握程度，为我生成一份从易到难的学习路径，以便我高效备战面试。
- **US-1.2**：作为一名学生，我希望每完成一个知识点后，系统自动出一道练习题测试我的掌握情况，以便我及时查漏补缺。
- **US-1.3**：作为一名学生，我希望系统能把知识点用"类比 + 代码示例"的方式讲解，因为纯文字理论我看不进去。

### 2.2 Persona 2：Lisa — 30 岁产品经理（职场学习者）

| 属性 | 描述 |
|------|------|
| 年龄 | 30 岁 |
| 目标 | 利用碎片时间学习 Python 数据分析，辅助日常工作 |
| 痛点 | 工作繁忙，每天只能学 30 分钟；网课太长坚持不下来 |
| 技术素养 | 初级，仅会 Excel 操作 |

**典型 User Story：**

- **US-2.1**：作为一名职场学习者，我希望系统能根据我每天可用的学习时长（30 分钟），将内容拆分为合适粒度的学习单元，以便我利用碎片时间完成学习。
- **US-2.2**：作为一名零基础学习者，我希望系统生成的资源优先使用通俗语言和实际案例，减少抽象术语，以便我更容易理解。
- **US-2.3**：作为一名学习者，我希望每周收到一份学习报告，了解我这周的进度和下周建议，以便保持学习动力。

### 2.3 Persona 3：王老师 — 高中数学教师

| 属性 | 描述 |
|------|------|
| 年龄 | 45 岁 |
| 目标 | 为不同层次的学生快速生成差异化的练习和讲义 |
| 痛点 | 一个班 50 个学生，水平参差不齐，无法为每人定制内容 |
| 技术素养 | 基础，能使用 Office 和简单 Web 应用 |

**典型 User Story：**

- **US-3.1**：作为一名教师，我希望输入一个知识点后，系统能同时生成"基础 / 进阶 / 挑战"三个难度等级的练习题集，以便我分层布置作业。
- **US-3.2**：作为一名教师，我希望查看班级整体的知识点掌握热力图，以便我精准定位教学薄弱环节。
- **US-3.3**：作为一名教师，我希望能导出生成的学习资源为 PDF / Word 格式，以便在课堂上分发使用。

> ⚠️ **假设**：教师角色为 P1 优先级，MVP 阶段以学生（自主学习者）为核心用户，教师功能在第二阶段迭代中加入。

#### F01：用户注册与基础信息收集

**优先级**：P0

**描述**：用户首次进入系统时，通过注册流程收集基础信息，为后续画像分析提供初始数据。

**输入**：
- 基础信息：昵称、邮箱（或手机号）、密码
- 学习偏好问卷（5-8 道题）：
  - 学习目标（如：备考 / 兴趣学习 / 技能提升）
  - 当前水平自评（入门 / 初级 / 中级 / 高级）
  - 每日可用学习时长（15min / 30min / 1h / 2h+）
  - 偏好的学习方式（视频 / 文字 / 实践 / 混合）
  - 感兴趣的学习科目（从预定义列表中选择）

**输出**：
- 用户账号创建成功
- 初始学习画像生成（传递给 ProfileAgent）

**验收标准**：
1. 注册流程在 3 步内完成（基础信息 → 偏好问卷 → 完成）
2. 偏好问卷支持跳过，系统使用默认值填充
3. 注册完成后 2 秒内跳转至学习首页
4. 支持邮箱 + 密码注册；⚠️ **假设**：MVP 暂不支持第三方登录（微信 / Google），待确认

---

#### F02：学习画像建立与更新

**优先级**：P0  
**对应 Agent**：ProfileAgent

**描述**：基于用户注册信息 + 持续学习行为数据，建立并动态更新用户的多维学习画像。

**画像维度**：

| 维度 | 数据来源 | 更新频率 |
|------|---------|---------|
| 知识水平（按知识点） | 评估结果 + 练习正确率 | 每次评估后 |
| 学习风格偏好 | 注册问卷 + 资源交互数据 | 每周 |
| 学习节奏 | 登录频率 + 单次学习时长 | 每日 |
| 薄弱知识点 | 评估错误分析 | 每次评估后 |
| 学习目标与动机 | 注册问卷 + 用户主动修改 | 用户触发 |

**输入**：
- 用户注册信息
- 学习行为事件流（页面停留时长、资源点击率、练习作答数据）
- EvaluatorAgent 的评估结果

**输出**：
- 结构化用户画像 JSON（供其他 Agent 消费）
- 画像变更事件（触发路径 / 资源重新生成）

**验收标准**：
1. 新用户注册后 3 秒内完成初始画像生成
2. 画像数据通过 API 可查，返回格式为标准 JSON Schema
3. 画像更新后，系统在下一次资源推荐时使用最新画像
4. 用户可在个人设置中查看并手动修正画像标签（如：将"初级"改为"中级"）

---

#### F03：个性化学习资源生成

**优先级**：P0  
**对应 Agent**：ResourceGenAgent

**描述**：根据用户画像和当前学习进度，实时生成匹配用户水平和偏好的学习资源。

**支持的资源类型**：

| 资源类型 | 描述 | MVP 支持 |
|---------|------|---------|
| 知识讲解 | 针对单个知识点的结构化讲解（支持多种表述风格） | 是 |
| 练习题 | 选择题 / 填空题 / 编程题，带详细解析 | 是 |
| 思维导图文本 | 知识点关联的结构化大纲（可渲染为思维导图） | 是 |
| 案例分析 | 基于真实场景的知识应用案例 | 否（P1） |
| 闪卡（Flashcard） | 适合碎片化记忆的问答卡片 | 否（P1） |

**输入**：
- 用户画像（来自 ProfileAgent）
- 当前学习路径节点（来自 PathPlannerAgent）
- 用户的即时请求（如："帮我讲讲快速排序"）

**输出**：
- 结构化学习资源（Markdown 格式）
- 资源元数据（难度等级、预计学习时长、关联知识点）

**验收标准**：
1. 同一知识点，不同画像的用户生成的资源内容存在明显差异（如语言风格、示例选取）
2. 知识讲解类资源生成时间 < 8 秒（首 token 延迟 < 2 秒，采用流式输出）
3. 练习题生成时附带标准答案和解析
4. 资源内容准确性由评估模块抽检，事实性错误率 < 5%
5. 用户可对生成资源进行"有用 / 没用"反馈，反馈数据回流至画像

> ⚠️ **假设**：资源生成完全由 LLM 实时生成，不依赖预制资源库。若后续发现高频知识点重复生成导致成本过高，考虑引入缓存机制。

#### F04：学习路径智能规划

**优先级**：P0  
**对应 Agent**：PathPlannerAgent

**描述**：基于用户画像、目标科目的知识图谱和学习进度，生成个性化的学习路径，并根据评估反馈动态调整。

**输入**：
- 用户画像（来自 ProfileAgent）
- 科目知识图谱（预定义的知识点依赖关系）
- 用户选定的学习目标（如："掌握数据结构基础"）
- 评估结果（来自 EvaluatorAgent，用于路径调整）

**输出**：
- 学习路径树（有序的知识点列表，含分支和前置依赖）
- 每个节点的预估学习时长
- 路径调整建议（如："建议先回顾链表，再学习树"）

**验收标准**：
1. 用户选定目标后，3 秒内生成完整学习路径
2. 路径中的知识点顺序尊重前置依赖关系（如：不会在用户未学"变量"前安排"函数"）
3. 当评估发现用户某知识点未达标时，自动在路径中插入补强节点
4. 用户可手动调整路径（跳过已掌握的节点、添加额外节点）
5. 路径可视化展示为时间线或树状结构

> ⚠️ **假设**：知识图谱初期由产品团队人工维护（预定义 JSON 格式），后续迭代中考虑自动化构建。需确认首发科目的知识图谱谁来提供。

#### F05：学习效果实时评估

**优先级**：P0  
**对应 Agent**：EvaluatorAgent

**描述**：在学习过程中的关键节点自动触发评估，分析学习效果，输出评估报告并驱动路径调整。

**评估触发时机**：
- 每个知识点学习完成后（mini quiz，3-5 道题）
- 学习路径阶段完成后（阶段测试，10-15 道题）
- 用户主动发起测试请求

**输入**：
- 评估题目与用户作答数据
- 用户学习行为数据（资源停留时长、重复阅读次数）
- 用户画像（用于题目难度匹配）

**输出**：
- 单次评估报告（得分、知识点掌握度、错误分析）
- 画像更新建议（传递给 ProfileAgent）
- 路径调整建议（传递给 PathPlannerAgent）

**验收标准**：
1. 评估题目难度与用户当前水平匹配（基于画像），不出现"碾压式"难题或"无意义"简单题
2. 评估结果在用户提交答案后 3 秒内返回
3. 每道错题附带错误原因分析和推荐复习知识点
4. 评估数据实时更新用户画像中的知识掌握度
5. 阶段测试结束后自动生成评估摘要，包含进步趋势可视化

#### F06：智能对话交互（学习助手）

**优先级**：P0  
**对应 Agent**：OrchestratorAgent

**描述**：用户通过自然语言对话与系统交互，OrchestratorAgent 理解用户意图并调度相应 Agent 完成任务。

**支持的意图类型**：

| 意图类别 | 示例用户输入 | 调度目标 |
|---------|------------|---------|
| 学习请求 | "帮我讲讲二叉树" | ResourceGenAgent |
| 路径查询 | "我接下来该学什么" | PathPlannerAgent |
| 评估请求 | "测试一下我对排序的掌握" | EvaluatorAgent |
| 画像修改 | "我觉得我已经掌握链表了" | ProfileAgent |
| 进度查询 | "我这周学了多少" | 内部数据查询 |
| 闲聊 / 鼓励 | "我学不下去了" | OrchestratorAgent（直接回复） |

**输入**：
- 用户自然语言消息
- 当前会话上下文

**输出**：
- 自然语言回复（流式输出）
- 可能附带的结构化内容（资源卡片、路径图、评估结果）

**验收标准**：
1. 意图识别准确率 > 90%（基于标注测试集）
2. 首 token 响应时间 < 1.5 秒
3. 对话支持多轮上下文理解（如用户说"再详细讲讲"，系统理解指代上一个知识点）
4. 对话界面支持 Markdown 渲染（代码块、表格、公式）
5. 当意图不明确时，主动追问而非猜测执行

---

#### F07：学习进度仪表盘

**优先级**：P1

**描述**：可视化展示用户的学习进度、知识掌握情况和学习趋势。

**包含模块**：
- **学习路径进度条**：当前路径完成百分比 + 预计剩余时间
- **知识掌握雷达图**：按知识点维度展示掌握程度（0-100%）
- **学习时长统计**：日 / 周 / 月学习时长趋势图
- **连续学习天数**：打卡日历

**输入**：
- 用户学习行为数据
- 评估结果数据

**输出**：
- 可视化仪表盘页面

**验收标准**：
1. 仪表盘页面加载时间 < 1.5 秒
2. 数据实时更新（用户完成一次学习后刷新即可看到最新数据）
3. 支持移动端响应式展示

#### F08：学习周报与成就系统

**优先级**：P1

**描述**：每周自动生成学习总结报告，并通过成就徽章激励用户持续学习。

**周报内容**：
- 本周学习时长与知识点完成数
- 进步最大的知识点 & 仍需加强的知识点
- 下周学习建议
- 与上周的对比趋势

**成就系统**：
- "连续学习 7 天"、"完成 100 道练习题"、"首次满分评估" 等成就徽章
- 成就解锁时弹窗提醒

**验收标准**：
1. 周报在每周一 08:00 自动生成并推送通知
2. 成就徽章实时解锁，弹窗延迟 < 1 秒
3. 用户可在个人中心查看历史周报和已获得的成就

> ⚠️ **假设**：通知渠道 MVP 阶段仅支持站内消息，后续考虑邮件 / 微信推送。

---

#### F09：教师工作台

**优先级**：P1

**描述**：为教师角色提供批量资源生成和班级学情管理功能。

**功能模块**：
- **批量资源生成**：输入知识点 + 难度等级，一键生成多份差异化练习 / 讲义
- **班级管理**：创建班级、邀请学生、查看班级整体学情
- **学情热力图**：按知识点展示全班掌握情况，快速定位教学薄弱点

**验收标准**：
1. 批量生成 3 个难度等级的练习题集耗时 < 15 秒
2. 班级支持最多 100 名学生
3. 学情热力图数据每日更新一次

---

#### F10：资源导出

**优先级**：P1

**描述**：用户可将系统生成的学习资源导出为离线可用的文件格式。

**支持格式**：PDF、Markdown、Word（.docx）

**验收标准**：
1. 导出操作在 5 秒内完成（单个资源）
2. 导出文件保留原有格式（标题层级、代码块高亮、表格）
3. 支持批量导出（选择多个资源一键打包为 ZIP）

## 4. 多智能体交互流程

### 4.1 主流程：用户发起学习

```
用户发送消息（如："我要开始学习数据结构"）
       │
       ▼
 ┌─────────────────┐
 │ OrchestratorAgent│  ← 意图识别 & 任务分解
 └────────┬────────┘
          │
          ├──① 查询/创建用户画像 ──▶ ProfileAgent
          │                              │
          │◀── 返回用户画像 JSON ────────┘
          │
          ├──② 生成/获取学习路径 ──▶ PathPlannerAgent
          │       （携带用户画像            │
          │        + 知识图谱）             │
          │◀── 返回学习路径 ──────────────┘
          │
          ├──③ 生成当前节点资源 ──▶ ResourceGenAgent
          │       （携带用户画像            │
          │        + 当前路径节点）          │
          │◀── 返回学习资源（流式）────────┘
          │
          ▼
    组装回复，流式输出给用户
    （学习资源 + 路径概览 + 引导语）
```

**流程说明**：

1. **OrchestratorAgent** 接收用户消息，执行意图识别。判定为"学习请求"后进入学习主流程。
2. 调用 **ProfileAgent** 获取用户最新画像。若用户为新用户且未完成画像初始化，先引导用户完成偏好问卷。
3. 将画像传递给 **PathPlannerAgent**，请求生成或获取已有的学习路径。PathPlannerAgent 基于知识图谱和用户水平规划路径。
4. 确定当前应学习的路径节点后，调用 **ResourceGenAgent** 生成个性化学习资源。
5. OrchestratorAgent 将资源 + 路径信息组装为用户友好的回复，流式输出。

### 4.2 评估与自适应调整流程

```
用户完成知识点学习
       │
       ▼
 OrchestratorAgent 自动触发评估
       │
       ├──▶ EvaluatorAgent
       │      │
       │      ├─ 生成评估题目（基于画像难度）
       │      ├─ 收集用户作答
       │      ├─ 分析结果
       │      │
       │      ├──▶ ProfileAgent（更新画像：知识掌握度）
       │      │
       │      └──▶ PathPlannerAgent（路径调整建议）
       │              │
       │              ├─ 掌握良好 → 继续下一节点
       │              ├─ 部分掌握 → 插入补充练习
       │              └─ 未掌握 → 回退至前置知识点
       │
       ▼
 OrchestratorAgent 输出评估结果 + 下一步建议
```

### 4.3 关键决策点

| 决策点 | 触发条件 | 决策逻辑 | 结果 |
|--------|---------|---------|------|
| 是否需要初始化画像 | 新用户首次发起学习 | ProfileAgent 检测无画像记录 | 引导进入问卷流程 |
| 资源难度选择 | ResourceGenAgent 生成资源时 | 基于画像中知识掌握度：<40% → 入门，40-70% → 进阶，>70% → 挑战 | 调整资源难度参数 |
| 路径节点推进 vs 回退 | 评估完成后 | 掌握度 ≥ 70% → 推进；40-70% → 补充练习；<40% → 回退 | 动态修改路径 |
| 意图分类兜底 | OrchestratorAgent 无法识别意图 | 置信度 < 0.6 | 向用户追问澄清 |
| 资源重新生成 | 用户反馈"没用" | 同一知识点连续 2 次负面反馈 | 切换讲解风格重新生成 |

### 4.4 异常处理

| 异常场景 | 处理策略 |
|---------|---------|
| LLM 调用超时（> 15 秒） | 返回友好提示"正在努力思考中..."，自动重试 1 次。若仍超时，降级返回缓存资源（如有）或提示用户稍后再试 |
| Agent 间通信失败 | OrchestratorAgent 捕获异常，记录日志，向用户返回"系统暂时开小差了，请稍后再试" |
| 知识图谱缺失（用户请求的科目未配置） | 返回提示"该科目正在建设中"，引导用户选择已支持的科目 |
| 用户输入包含敏感/不安全内容 | OrchestratorAgent 内置内容安全过滤，拦截并提示用户调整输入 |
| 画像数据异常（字段缺失/格式错误） | ProfileAgent 使用默认值填充并记录告警日志 |

## 5. 非功能需求

### 5.1 性能要求

| 指标 | 要求 | 说明 |
|------|------|------|
| 对话首 token 延迟 | < 1.5 秒 | 从用户发送消息到首个字符出现 |
| 资源生成完成时间 | < 8 秒（流式） | 完整知识点讲解 |
| 评估结果返回 | < 3 秒 | 用户提交答案到结果展示 |
| 页面加载时间 | < 2 秒 | 首屏加载（含 SSR / CDN） |
| 并发用户数 | MVP：500 同时在线 | 后续按需扩展至 5000+ |
| LLM API 调用 QPS | MVP：50 QPS | 需做请求队列和限流 |

### 5.2 安全与隐私

| 类别 | 要求 |
|------|------|
| 数据传输 | 全站 HTTPS，WebSocket 加密 |
| 用户认证 | JWT Token + Refresh Token 机制 |
| 密码存储 | BCrypt 加密，不存储明文 |
| LLM 交互安全 | 输入内容安全过滤（敏感词、Prompt 注入防护） |
| 数据隐私 | 符合《个人信息保护法》，用户画像数据不对外共享 |
| 数据删除 | 用户注销时，30 天内删除所有个人数据 |
| 日志脱敏 | 日志中不记录用户原始输入内容，仅记录意图分类和元数据 |

> ⚠️ **假设**：系统部署在中国大陆，需符合国内法规。若面向海外用户，需额外遵守 GDPR。

### 5.3 可扩展性

| 方面 | 设计要求 |
|------|---------|
| 科目扩展 | 新增科目仅需配置知识图谱 JSON + Prompt 模板，无需修改核心代码 |
| Agent 扩展 | 基于 LangGraph 的模块化架构，新增 Agent 只需实现标准接口并注册到 OrchestratorAgent |
| LLM 模型切换 | 支持通过配置切换底层 LLM（如 GPT-4o → Claude → 国产模型），不耦合特定模型 API |
| 多语言支持 | 前端国际化框架（Vue i18n），后端消息模板化 |
| 水平扩展 | Spring Boot 服务无状态化，支持多实例部署；LangGraph 服务支持独立扩缩容 |

## 6. 数据需求

### 6.1 数据采集与存储

| 数据类别 | 具体字段 | 存储位置 | 保留策略 |
|---------|---------|---------|---------|
| **用户基础信息** | 昵称、邮箱、注册时间、角色（学生/教师） | MySQL | 账号存续期间 |
| **学习画像** | 知识掌握度矩阵、学习风格、偏好参数 | MySQL + Redis 缓存 | 实时更新，保留历史快照（月级） |
| **学习行为日志** | 页面访问、资源点击、停留时长、滚动深度 | ClickHouse / MongoDB | 原始日志保留 90 天，聚合数据永久 |
| **评估数据** | 题目 ID、用户答案、正确答案、耗时、评分 | MySQL | 永久保留 |
| **对话记录** | 会话 ID、用户消息摘要、意图分类、Agent 调度链 | MySQL + 对象存储 | 完整记录保留 30 天，摘要永久 |
| **生成资源缓存** | 知识点 ID + 画像哈希 → 资源内容 | Redis + 对象存储 | LRU 淘汰，热门资源常驻 |
| **反馈数据** | 资源 ID、反馈类型（有用/没用）、文字评价 | MySQL | 永久保留 |

> ⚠️ **假设**：对话中的用户原始消息仅保留 30 天（隐私合规），之后仅保留脱敏后的摘要和意图分类。需法务确认保留策略。

### 6.2 数据驱动个性化

```
                    数据采集
                       │
        ┌──────────────┼──────────────┐
        │              │              │
   学习行为日志    评估作答数据    用户反馈
        │              │              │
        └──────┬───────┴──────┬───────┘
               │              │
               ▼              ▼
        ProfileAgent     EvaluatorAgent
        (画像更新)       (效果分析)
               │              │
               └──────┬───────┘
                      │
                      ▼
              ┌───────────────┐
              │  用户画像 JSON  │
              └───────┬───────┘
                      │
           ┌──────────┼──────────┐
           │                     │
           ▼                     ▼
    PathPlannerAgent      ResourceGenAgent
    (路径调整)            (资源适配)
           │                     │
           └──────────┬──────────┘
                      │
                      ▼
               个性化学习体验
```

**关键数据驱动逻辑**：

1. **知识掌握度建模**：基于评估正确率 + 遗忘曲线模型（间隔重复），动态计算每个知识点的掌握概率。当概率衰减至阈值以下时，自动安排复习。
2. **学习风格适配**：统计用户对不同类型资源的完读率和反馈评分，自动调整资源生成的风格权重（如：代码示例 60% + 图示 30% + 文字 10%）。
3. **学习节奏预测**：基于历史登录和学习时长数据，预测用户偏好的学习时段和单次学习时长，据此调整每日推荐内容量。
4. **难度自适应**：评估结果驱动实时难度调节——连续 3 次高分自动提升难度，连续 2 次低分自动降低难度并插入基础巩固。

## 7. 里程碑规划

### 7.1 MVP（v1.0）— 最小可行产品

**目标**：验证"AI 个性化学习闭环"核心价值假设

**范围**：

| 模块 | 包含内容 |
|------|---------|
| 用户系统 | 邮箱注册 + 登录 + 偏好问卷（F01） |
| 学习画像 | 基础画像建立与自动更新（F02） |
| 对话交互 | 自然语言学习助手，支持核心意图识别（F06） |
| 资源生成 | 知识讲解 + 练习题，支持流式输出（F03） |
| 路径规划 | 基于知识图谱的自动路径生成 + 基础调整（F04） |
| 学习评估 | 知识点 mini quiz + 结果分析 + 画像更新联动（F05） |
| 科目范围 | 计算机科学（数据结构与算法） |

**不包含**：教师功能、导出功能、仪表盘可视化、成就系统、社区功能

> ⚠️ **假设**：MVP 开发周期预计 8-10 周（含前后端 + Agent 开发 + 联调测试）。需确认团队规模和实际排期。

- **AI 助教**：教师可训练专属 AI 助教，自动回答学生常见问题
- **协作学习**：支持学习小组内的 AI 辅助协作任务（如：结对编程练习）
